Contributions are welcome! Recur is an open-source project, and we appreciate contributions from the community to make it better.

## Getting started

Before contributing, make sure you have set up your development environment by following the [setup guide](/development/setup).

<Steps>
  <Step title="Fork and clone the repository">
    Fork the repository on GitHub and clone it to your local machine:
    
    ```bash
    git clone https://github.com/YOUR_USERNAME/recur.git
    cd recur
    ```
  </Step>

  <Step title="Create a new branch">
    Create a feature branch for your changes:
    
    ```bash
    git checkout -b feature/your-feature-name
    ```
  </Step>

  <Step title="Make your changes">
    Make your changes following the code style guidelines below.
  </Step>

  <Step title="Test your changes">
    Test your changes thoroughly on both iOS and Android platforms:
    
    ```bash
    bun dev
    # Press 'i' for iOS or 'a' for Android
    ```
  </Step>

  <Step title="Commit and push">
    Commit your changes with a clear, descriptive commit message:
    
    ```bash
    git add .
    git commit -m "Add feature: your feature description"
    git push origin feature/your-feature-name
    ```
  </Step>

  <Step title="Open a pull request">
    Open a pull request on GitHub with a description of your changes.
  </Step>
</Steps>

## Code style

Recur uses [Prettier](https://prettier.io/) for code formatting. The project includes a `.prettierrc` configuration:

```json
{
  "printWidth": 100,
  "tabWidth": 2,
  "singleQuote": true,
  "bracketSameLine": true,
  "trailingComma": "es5",
  "plugins": ["prettier-plugin-tailwindcss"],
  "tailwindFunctions": ["cva"]
}
```

### Style guidelines

- **TypeScript**: Use TypeScript for all new code
- **Single quotes**: Use single quotes for strings
- **Line length**: Maximum 100 characters per line
- **Indentation**: 2 spaces (no tabs)
- **Trailing commas**: Use ES5 trailing commas
- **NativeWind**: Use NativeWind classes for styling, not inline styles
- **Zod validation**: Use Zod schemas for runtime validation in actions

<Tip>
  Run Prettier before committing to ensure your code follows the style guidelines:
  
  ```bash
  npx prettier --write .
  ```
</Tip>

## Project conventions

### File organization

- **Actions**: Business logic goes in `actions/` directory with Zod validation
- **Components**: Reusable UI components go in `components/`
- **Screens**: App screens go in `app/` following Expo Router conventions
- **Hooks**: Custom React hooks go in `hooks/`
- **Types**: Shared TypeScript types go in `types.ts`
- **Constants**: App constants go in `const.ts`

### Naming conventions

- **Files**: Use kebab-case for file names (e.g., `add-subscriptions-dialog.tsx`)
- **Components**: Use PascalCase for component names (e.g., `AddSubscriptionsDialog`)
- **Functions**: Use camelCase for function names (e.g., `createSubscription`)
- **Types**: Use PascalCase for type names (e.g., `Subscription`, `List`)

### Data validation

All data operations must use Zod for runtime validation:

```typescript
import { z } from 'zod';

const Zod_MySchema = z.object({
  name: z.string().min(1),
  amount: z.number().nonnegative(),
});

export const myAction = (args: unknown) => {
  const validated = Zod_MySchema.safeParse(args);
  if (!validated.success) {
    logger.error('Invalid input', validated.error);
    throw new Error('Invalid input');
  }
  // Use validated.data
};
```
